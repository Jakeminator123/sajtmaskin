## Förändringar som sannolikt sänker kvaliteten

Här är de mest “load‑bearing” kodförändringarna som kan göra att generationer upplevs mindre imponerande, med exakta diff‑snuttar där det går.

### Prompt Assist/“Förbättra” ger ofta mindre berikning

I commit **`c4a46ba`** ändrades `formatPrompt()` så att den **tidigt returnerar** en kort, normaliserad prompt om det inte finns “extra” metadata och prompten är kort (<300). citeturn53view0

Nyckelsnutt (beteendeskifte):

```ts
const hasExtras = sections.length > 0 || styles.length > 0 || constraints.length > 0 || urls.length > 0;
if (!hasExtras && normalized.length < 300) return normalized;
```

Konsekvens: om din workflow ofta är “skriv en kort prompt → klicka Förbättra → generera”, så kan generatorn nu få **mindre explicit layout/sektioner/visuell riktning** än tidigare. Det brukar ge mer generiska sidor.

Dessutom ändrades “rewrite system prompt”‑reglerna så att output *inte* ska wrap:as i generiska rubriker som “Mål/Sektioner/Stil/Constraints”, och att en kort input ska ge kort output — vilket är bra för kostnad/latency men kan minska “wow‑faktor” om du förväntar dig att assistenten fyller i mer design. citeturn53view0

### Tokenbudget och reasoning‑koppling kan göra output kortare eller mindre kapabel

Commit **`58b4962`** införde:
- per‑tier `maxOutputTokens` (fast/pro/max/codex/anthropic) och
- koppling mellan “Thinking” och OpenAI “reasoningEffort” via `providerOptions`, samt
- att `modelTier` nu skickas in i `generateCode`. citeturn20view1

Kod i `engine.ts` visar mekaniken (förkortat):

```ts
const resolvedMaxTokens = maxTokens ?? getEngineMaxOutputTokens(modelTier);
const reasoningEffort = getReasoningEffort(modelTier, thinking);

const result = streamText({
  ...
  maxOutputTokens: resolvedMaxTokens,
  ...(providerOptions ? { providerOptions } : {}),
});
```

citeturn20view1

Varför detta kan ge regression:
- Om du tidigare hade en **hög env‑override** (t.ex. högt `SAJTMASKIN_ENGINE_MAX_OUTPUT_TOKENS`) och nu alltid skickar `modelTier`, kan den override i praktiken spela mindre roll eftersom per‑tier‑tabellen används när tier finns (och bara “fallbackar” när tier saknas). Det framgår också i dokumentationsdiffen att tiers har egna tak och att env‑värdet används som fallback. citeturn20view1turn53view0
- Om du ofta kör tier “fast”, så är reasoningEffort för fast definierad som `"none"` även när Thinking är på (enligt den nya tier‑mappingen i defaults). Då blir resultatet typiskt mindre “resonerande”/ambitiöst än om du kör “pro/max/codex”. citeturn20view1turn53view0

### Modellmappningar ändrades och codex‑profilen bytte defaultmodell

I **`c4a46ba`** flyttades “codex” default från `gpt-5.1-codex-max` till `gpt-5.4`, både i default‑profil och i katalog/mappning. citeturn53view0

Det här kan påverka upplevd kvalitet på två sätt:
- Om `gpt-5.1-codex-max` gav dig en särskild “kod‑stil” eller mer aggressiv kodproduktion, kan `gpt-5.4` kännas annorlunda.
- Kvalitetskartan (`QUALITY_TO_OPENAI_MODEL`) ändrades så att `max` går till `gpt-5.4` i stället för `gpt-5.1-codex-max`. citeturn53view0

Det är alltså fullt möjligt att du “kör samma tier i UI” men i praktiken fått en annan modell bakom kulisserna.

### Scaffoldvalet har ändrats (vilket ofta känns som “kvalitet”)

I **`941ddba`** ändrades scaffolds/taggar/manifest så att `content-site` inte längre beskrivs som “hero/features/testimonials” för landing/portfolio/blog, utan som en dokumentations-/informations‑layout med sidebar och FAQ‑känsla. Dessutom rensades tags som “landing/portfolio/blog/marketing” bort och ersattes av mer “docs/knowledge base”‑taggar. citeturn49view0turn50view0

Det betyder att prompts som tidigare råkade matcha `content-site` kan matcha **annat scaffold** nu (t.ex. `landing-page`), vilket kan kännas mer generiskt och därmed mindre imponerande.

Samtidigt introducerades nya scaffolds (clinic/local-shop/event/school). Det kan vara förbättringar i sak, men i en övergångsperiod är det vanligt att matchningen “flappar” (byter scaffold för samma prompt) och då får du stora skillnader i layoutkvalitet. citeturn50view0turn49view0

Utöver detta finns en mycket stor “scaffold system overhaul”‑commit (**`b412991`**) som explicit säger att den förändrar pipeline‑struktur, lägger nya scaffolds, förbättrar svensk matchning, och visar match‑metadata i UI. Den typen av commit är nästan alltid en kandidat när output “känns annorlunda” även om själva LLM‑prompten är samma. citeturn40view0turn44view0

### Direct-provider refactor kan ändra assist/brief-fasen som formar generationen

I **`ef71502`** byttes `gateway(normalizedModel)` ut mot `createDirectModel(normalizedModel)` i flera route‑endpoints (bl.a. `/api/ai/chat` och `/api/ai/brief`). Samtidigt togs gateway‑specifika fallback‑listor och provider‑order‑routing bort från requesten. citeturn23view0

Även om “own engine” kör via `src/lib/gen/engine.ts`, så påverkas totalen av att prompten ofta först passerar “prompt assist” och “deep brief”. Om de stegen blir mer sparsamma, eller tappar fallback, kan slutprompten bli sämre → sämre generation.

### Anthropic-brief schema förenklades (stabilitet vs detaljrikedom)

I **`369b95f`** togs flera `min()`‑krav bort (t.ex. `toneAndVoice` min 2, bullets min 1, styleKeywords min 3, etc.) för att hålla sig inom Anthropic’s grammar‑kompilationsgränser, och förenklade schemat gjordes mindre “optional‑tungt”. citeturn51view0

Detta kan betyda att brief‑agenten (särskilt i Claude‑läge) nu *kan* returnera mer tomma listor eller färre punkter, vilket kan kännas mindre “rikt”. (Det kan samtidigt vara helt rätt tradeoff om du tidigare hade hårda failures.)
