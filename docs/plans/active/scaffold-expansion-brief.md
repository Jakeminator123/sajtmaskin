### 1.4 Pipelinens steg

```
1. Discovery     npm run references:discover         (Playwright, skrapar vercel.com/templates)
2. Import        npm run template-library:import-legacy  (normaliserar till discovery/current/)
3. Hydrate       npm run template-library:hydrate-cache  (shallow clones)
4. Build         npm run template-library:build          (dossiers + generated JSON)
5. Embeddings    npm run template-library:embeddings     (OpenAI vektorer for template-library)
6. Scaffold emb  npm run scaffolds:embeddings            (OpenAI vektorer for runtime-scaffolds)
7. Test          npm run scaffolds:test-matching         (90+ testprompter)
```

Interaktiv meny: `python scaffold-pipeline/scripts/scaffold-pipeline.py`
Snabbkommando: `npm run scaffold-pipeline` (steg 3-7) eller
`npm run scaffold-pipeline:full` (steg 1+3-7).

## 5. Sakerhet vid utvidgning — non-Next.js-content

### 5.1 Problemet

Vercel templates innehaller manga ramverk: Nuxt, Svelte, Astro, Remix,
Hono, Express, Gatsby, Angular, m.fl. Sajtmaskin genererar ENBART
Next.js-applikationer. Om pipeline-utvidgningen okritiskt importerar
icke-Next-templates kan det:

1. Skapa dossiers med irrelevanta kodutdrag (Vue-komponenter, Svelte-stores)
2. Forstora template-library.generated.json med entries som aldrig matchas
3. Injicera missvisande "Reference Code Snippets" i system prompt
4. Ge hogre embeddings-kostnad utan nyttovinst

### 5.2 Befintligt skydd

`build-template-library.ts` har redan en verdict-logik som klassificerar
templates. Verdict `non_next_template` anvands for 29 av 105 auditerade.
Dessa filtreras bort fran kuraterade entries.

Matchern (`matcher.ts`) matchar enbart mot de 13 interna scaffolds i
registry.ts. Allt som inte ar Next.js-relevant nar aldrig genererings-
prompten.

### 5.3 Rekommendation for sakert utvidgande

1. **Utvidga slug-listan i CLI:n** — ingen risk, vi far mer raadata.
2. **Lat build-scriptet filtrera** — `non_next_template` verdict gallrar
   automatiskt. Inga manuella steg kravs.
3. **Monitorera kvalitet** — kor `npm run scaffolds:test-matching` efter
   varje utvidgning for att verifiera att matchningsresultat inte
   forsamras.
4. **Overvaeg research-only-flag** — for icke-Next templates som anda har
   intressanta UX-monster (t.ex. Astro-baserade portfolios) kan man
   behalla dem med verdict `research_only` utan att de paverkar runtime.
5. **Undvik att skapa nya ScaffoldFamily-varden** om det inte ar absolut
   nodvandigt. Lat nya scaffolds ateranvanda befintliga familjer (som
   restaurant -> landing-page). Det haaller types.ts stabilt och minskar
   embeddings-drift.

### 5.4 Nar en ny ScaffoldFamily faktiskt behovs

Lagg till en ny family ENBART nar:
- Befintliga familjer saknar den arkitektoniska grundstrukturen
- Minst 3-5 svenska prompter i testmatrien skulle fa fel scaffold med
  nuvarande familjer
- Ny family har minst 3 unika filer som inte ar varianter av befintliga

Exempel: "clinic" (mottagning) ar troligen INTE en ny family — den ar en
variant av landing-page eller booking. Men "marketplace" (marknadsplats
med saljare + kopare + admin) KAN vara en ny family om befintliga inte
tacker den dubblsidiga strukturen.

## 6. Forslag: template-search MCP-server

### 6.1 Vad den loser

Idag finns tre isolerade datakallar utan sokning tvarsom:

A) Extern template-data (dossiers + template-library.generated.json)
B) Anvandarnas genererade kod (sajtmaskin-engine MCP, kraver chatId)
C) Anvandarnas deployade sidor (Vercel MCP, kraver teamId + projectId)

Ingen agent kan fraga: "vilka e-handel-templates finns?" eller "jamfor
vara restaurant-sidor med externa templates" utan att manuellt veta ratt
ID:n.

### 6.2 Foreslagen serverstruktur

Ny fil: `tools/mcp/template-search-server.ts`

Samma monster som `tools/mcp/engine-server.ts`:
- Anvand `@modelcontextprotocol/sdk` + StdioServerTransport
- Registrera i `.cursor/mcp.json`

### 6.3 Foreslaagna verktyg

```
search_templates
  Input: query (string), topK (number, default 5)
  Logik: anvand befintlig searchTemplateLibrary() fran
         src/lib/gen/template-library/search.ts
  Output: matchade entries med titel, kategori, poang, styrkor

list_template_categories
  Input: (inget)
  Logik: las template-library.generated.json, gruppera per categorySlug
  Output: lista av kategorier med antal templates per kategori

get_template_dossier
  Input: templateId (string)
  Logik: las manifest.json + summary.md fran scaffold-pipeline/dossiers/
  Output: fullstandig dossierdata

compare_template_to_scaffold
  Input: templateId (string), scaffoldId (string)
  Logik: hamta dossier + scaffold manifest, jamfor struktur
  Output: likheter, skillnader, rekommendationer

list_scaffolds
  Input: (inget)
  Logik: anropa getAllScaffolds() fran registry.ts
  Output: alla 13 scaffolds med id, family, label, tags

search_scaffolds_by_prompt
  Input: prompt (string)
  Logik: anropa matchScaffold() fran matcher.ts
  Output: basta scaffold-match med kalla (keyword/embedding) och poang
```

### 6.4 Implementation — uppskattat arbete

- Grundserver (5 verktyg): ~200 rader TypeScript, 2-3 timmar
- Lagg till i `.cursor/mcp.json`:
  ```json
  "sajtmaskin-templates": {
    "command": "npx",
    "args": ["tsx", "tools/mcp/template-search-server.ts"]
  }
  ```
- Ingen ny dependency — ateranvander befintliga moduler

### 6.5 Framtida utvidgning

Fas 2 (efter grundservern):
- `search_user_sites` — sok bland anvandarnas genererade sidor via
  engine-databasen
- `get_site_comparison` — jamfor en genererad sajt med en extern template
- `suggest_scaffold_for_prompt` — rekommendation med forklaring

## 9. Kontaktyta fran runtime till pipeline

### 9.1 Hur template-library anvands vid generering

1. `system-prompt.ts` anropar `rankTemplateReferences(prompt, scaffold)`
2. Det anropar `searchTemplateLibrary(prompt)` som gor keyword + embedding-sok
3. Top 3 matchade entries injiceras som "Relevant Template References" i
   dynamiska system-prompten
4. Top 2 far ocksa sina `selectedFiles` (kodexcerpts) injicerade som
   "Reference Code Snippets"
5. Sokningen har MIN_EMBEDDING_SCORE = 0.3 och DEFAULT_TOP_K = 3
6. Max reference-filer: 20, max excerpt-chars: 9000, max totalt: 18000

### 9.2 Hur scaffolds matchas

1. `orchestrate.ts` kallar matchern (auto/manual/off)
2. `matcher.ts` gor keyword-sokning forst (LANDING_KEYWORDS, SAAS_KEYWORDS,
   PORTFOLIO_KEYWORDS, BLOG_KEYWORDS, etc. — alla pa bade engelska och
   svenska inkl. SNI-branschkoder)
3. Om keyword ger generisk match (base-nextjs) sa faller den tillbaka
   till embedding-sokning via `scaffold-search.ts`
4. `scaffold-scoring.ts` kan boosta/straffa baserat pa telemetri

### 9.3 Gallrande grans — vad som INTE nar runtime

- Templates med verdict `non_next_template`, `bad_repo_link`, eller
  `huge_monorepo` kurateras INTE och hamnar aldrig i generated JSON
- Templates med qualityScore under en troskelvard kurateras inte heller
- selectedFiles som saknas (t.ex. efter fresh checkout utan pipeline-kor)
  ger tomma kodexcerpts — systemet fungerar anda, bara utan snippets
- ScaffoldFamily-varden som inte finns i types.ts avvisas av TypeScript-
  kompilatorn innan de kan commitas
