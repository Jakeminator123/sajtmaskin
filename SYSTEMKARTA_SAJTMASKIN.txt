SAJTMASKIN - SYSTEMKARTA / MINDMAP
=================================

Syfte
-----
Detta dokument är en snabb, mänsklig översikt över:
- vad som är vad
- vilka lager som finns
- vad som används direkt av runtime
- vad som bara är intake / research / rebuild

3. Vad som faktiskt når LLM
---------------------------

Direkt input i own-engine-kedjan:
- användarens senaste meddelande (user-turn; skickas separat — dupliceras inte som extra "original request" i system)
- brief/spec (om finns)
- route plan (med provenance: primarySource + sources[])
- pre-generation contracts
- scaffold-kontext
- scaffold-traits
- theme / design references / media catalog
- custom instructions
- dynamisk kontext budgeteras/prunas; se DynamicContextPruning / docs/architecture/llm-input-blocks.md

Inte direkt input:
- hela externa repos
- fulla dossiers
- hela Sajtmaskin-repot


4. Scaffold-kontexten består av
-------------------------------

När ett scaffold väljs får modellen typiskt:
- scaffold description
- scaffold family
- scaffold traits
- file tree
- critical scaffold files
- quality checklist
- scaffold research priorities

Källor:
- src/lib/gen/scaffolds/registry.ts
- src/lib/gen/scaffolds/serialize.ts
- src/lib/gen/orchestrate.ts
- src/lib/gen/system-prompt.ts


5. BuildSpec / styrsignaler
---------------------------

Ord att kunna:
- buildIntent = vad som byggs
- generationMode = första build eller follow-up
- changeScope = vilken sorts ändring
- qualityTarget = kvalitetsnivå
- previewPolicy = previewnivå
- verificationPolicy = kontrollnivå efter build
- contextPolicy = light / normal / heavy
- routePlan.siteType = one-page / brochure / content-heavy / app-shell
- routePlan.provenance.primarySource = brief / prompt / scaffold
- persisted scaffold = tidigare scaffold-id som kan låsas upp vid tydlig redesign
- scaffoldFamily = scaffoldens runtime-bucket

Källa:
- src/lib/gen/build-spec.ts


6. light / normal / heavy
-------------------------

contextPolicy styr hur mycket kontext som skickas.

light
- små follow-ups
- copy / local-layout

normal
- standardfall

heavy
- appfall
- fler routes
- fler integrationer
- persisted/data-tyngre flöden


7. Post-codegen / Steg 4 (finalize)
----------------------------------

Efter own-engine codegen (stream) körs finalizeAndSaveVersion:
  src/lib/gen/stream/finalize-version.ts

Grovt flode:
  autofix -> url_expand -> validate_syntax -> (ev. materialize_images om deep path)
  -> (ev. verifier-pass om deep path + policy) -> parse_merge_preflight -> sparad version

Telemetry:
  - "fast+deep" = kör deep path (bildmaterialisering enligt policy)
  - "fast-only" = smal finalize (t.ex. lätt follow-up): hoppar över bildsteg och verifier i finalize

Asynk efter finalize (gräns till Steg 5):
  - server-verify / tier-2 quality gate: src/lib/gen/server-verify.ts

Viktiga gränssignaler runt preview:
  - previewPending = preview väntas efter finalize
  - previewUrlHint = temporär VM-hint i done-event
  - legacyShimPreviewUrl = äldre shim/fallback, inte primär tier-2-preview

Kanonisk karta:
  - docs/architecture/step4-post-generation.md
  - docs/plans/active/step4-quality-hotspots-and-verification.md

10. Rekommenderad nästa genomgång (bakåt mot aktiv LLM-kedja)
-------------------------------------------------------------

Steg 1
- Builder-entry, ingress, ID-termer

Steg 2
- Scaffold, route plan, BuildSpec, contracts

Steg 3
- Faktisk LLM-input: llm-input-blocks.md, DynamicContextPruning

Steg 4
- Finalize, validate, verifier, preflight, server-verify (se step4-post-generation.md)

Steg 5
- Preview, VM/preview_host, materialisering, uppföljning

Separat (inte samma som Steg 4/5 i kedjan):
- template-library / extern pipeline-städ när tid finns

Andra återkommande begrepp att hålla isär:
- entryKind = normaliserad builder-ingress
- externalProjectId = extern/legacy projektidentitet, inte appProjectId
- compat-routes / v0 streams = bakåtkompatibla HTTP-ytor, inte separat motor
- prompt-dumps = observability, inte source of truth
