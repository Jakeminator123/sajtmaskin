Buggrapport — allmänna potentiella buggar (Sajtmaskin, worktree hrx)
Genererad: 2026-03-27
Filnamn: slumpat fyrsiffrigt (5844.txt) för att minska kollision vid parallella agenter.

================================================================================
Projektöversikt (kort)
================================================================================
Sajtmaskin är en Next.js-app (App Router) med own-engine codegen under src/lib/gen/,
builder-SSE (src/lib/builder/sse.ts), deployment-händelser (src/app/api/v0/deployments/...)
och MCP-flöde generate-site (src/lib/mcp/generate-site.ts). Rapporten fokuserar på
strömmande läsning, felhantering och klienthooks där avvikelser syntes tydligast.

================================================================================
1. Planläge-strömmen ignorerar SSE-händelserna "error" och "done" från pipeline
================================================================================
Sannolikhet att det är en bugg: 85%

Motivering:
I src/lib/gen/stream/plan-mode-stream.ts hanteras case "done" och case "error" i
switch (evt.event) med enbart break, utan att avbryta läsloopen, vidarebefordra fel
till klienten eller sätta en flagga. Efter try/finally körs alltid resolvePlanArtifact,
persistAssistantSummary och ett "done"-event. Om upstream skickar ett explicit felflöde
via SSE-event "error" når det inte användaren; om "done" i SSE ska betyda slut kan
beteendet skilja sig från att enbart lita på att ReadableStream läses färdigt.

Påverkan på projektet: Hög — planläges-UX och kreditering kan triggas trots att
pipelinen signalerat fel; fel visas inte konsekvent.

================================================================================
2. MCP generate-site: "error"-event ignoreras om det redan finns partiellt innehåll
================================================================================
Sannolikhet att det är en bugg: 80%

Motivering:
I src/lib/mcp/generate-site.ts sätts streamError vid case "error", men kast sker bara
när !sawDone && !accumulatedContent.trim(). Om strömmen rapporterat fel men det finns
text i accumulatedContent fortsätter flödet till finalizeAndSaveVersion utan att
streamError stoppar. Det kan spara en partial eller inkonsistent version trots rapporterat
fel, om inte produktionspolicy uttryckligen tillåter det.

Påverkan på projektet: Hög för MCP-anropare som förväntar sig att fel innebär att ingen
giltig version sparas.

================================================================================
3. Deployment-status (klient): tom catch vid JSON.parse av EventSource
================================================================================
Sannolikhet att det är en bugg: 70%

Motivering:
I src/lib/hooks/useDeploymentStatus.ts omsluts JSON.parse(event.data) av try/catch med
tom catch. Felaktig eller oväntad payload ger inget felstate och ingen loggning; status
kan fortsätta vara "pending" utan att användaren förstår varför.

Påverkan på projektet: Medel — deploy-UI kan upplevas som "hängande" utan feedback vid
server-/formatfel.

================================================================================
4. Deployment SSE-route: tyst fel i Vercel-poll-loopen
================================================================================
Sannolikhet att det är en bugg: 65%

Motivering:
I src/app/api/v0/deployments/[deploymentId]/events/route.ts finns catch {} kring
hela poll-försöket mot getVercelDeployment. Vid upprepade API-fel skickas inga
uppdateringar till klienten; om Redis inte heller levererar händelser kan status
inte nå ett terminalt tillstånd.

Påverkan på projektet: Medel — deploy-övervakning i edge cases (API tillfälligt nere,
rate limits) utan synlig fallback-status.

================================================================================
5. consumeSseResponse: okända event:-namn släpps
================================================================================
Sannolikhet att det är en bugg: 55%

Motivering:
I src/lib/builder/sse.ts anropas onEvent endast om isBuilderStreamEventName(currentEvent).
Händelser med andra namn (t.ex. standard "message", stavfel eller framtida eventtyper)
parsas men vidarebefordras inte.

Påverkan på projektet: Låg till medel — medvetet protokollfilter kan vara avsiktligt;
blir problematiskt om backend utökar SSE utan att matcha kontraktsnamnen.
