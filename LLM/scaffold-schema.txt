SAJTMASKIN SCAFFOLD SCHEMA
Purpose: Describe the expected contract for internal scaffolds and how they should be used in generation.
Scope: Documentation only. This file defines conventions, not runtime validation.

WHY THIS FILE EXISTS
- The project already uses `LLM/` for standards and schema-style docs.
- The scaffold system is now important enough to need a stable written contract.
- This keeps scaffold work consistent across chats and implementations.

RELATION TO CODE
- Source of truth in runtime:
  - `src/lib/gen/scaffolds/types.ts`
  - `src/lib/gen/scaffolds/registry.ts`
  - `src/lib/gen/scaffolds/matcher.ts`
  - `src/lib/gen/scaffolds/serialize.ts`
- This file is a human-readable implementation contract.

SCaffold DEFINITION
- A scaffold is a small internal starter for one category of site or app.
- A scaffold is not a full external template import.
- A scaffold should be:
  - category-shaped
  - generic enough to reuse
  - strong enough to improve first output quality
  - safe for the model to modify

RECOMMENDED MANIFEST CONCEPT
Each scaffold should define at least:

1) Stable identity
- `id`: unique string
- `family`: broad grouping
- `label`: human-readable name
- `description`: short explanation of intended use

2) Selection hints
- `keywords`: prompt terms that suggest this scaffold
- `buildIntents`: compatible intents such as website, app, template
- `priority`: optional tie-break signal if multiple scaffolds match

3) Prompt serialization content
- `files`: starter files to inject into prompt context
- optional `notes`: compact instructions about how the model should treat the scaffold

4) File contract
Each file entry should include:
- `path`
- `content`
- optional `language`
- optional `purpose`

SCaffold QUALITY RULES
Every scaffold should:

- be minimal but opinionated
- include only files that meaningfully improve generation quality
- avoid dead code and placeholder complexity
- prefer reusable sections and patterns over one-off pages
- use the same stack assumptions as the engine

For website scaffolds:

- include a strong hero baseline
- include header/nav and footer unless there is a clear reason not to
- establish spacing rhythm and section hierarchy
- provide a visually credible starting shape

For app scaffolds:

- include a stable shell/layout
- establish navigation pattern
- include at least one realistic content area
- support iterative expansion without full rewrite

TECH STACK RULES
Scaffolds should align with the engine defaults:

- Next.js App Router
- TypeScript
- Tailwind CSS v4
- shadcn/ui usage through `@/components/ui/*`
- `lucide-react` for icons
- semantic token usage rather than arbitrary hardcoded styling

Do not make a scaffold depend on:

- external backend services
- secrets or env vars for first render
- large infra assumptions
- provider-specific commerce/auth stacks unless explicitly intended

FILE SHAPE GUIDELINES
Prefer a small number of high-leverage files.

Good examples:
- `app/page.tsx`
- `app/layout.tsx`
- `components/site-header.tsx`
- `components/site-footer.tsx`
- one or two key sectional components

Avoid:
- bloated multi-page starter trees unless the category truly needs it
- test/config/storybook baggage in first-wave site scaffolds
- copying a full third-party repository structure

SERIALIZATION RULE
When a scaffold is selected:

- the scaffold should be serialized as context before generation
- the model should treat scaffold files as editable starting files
- generated output may replace, extend, or refine scaffold files
- final saved version may merge scaffold files with generated files

MATCHING RULE
Scaffold matching should consider:

- build intent
- prompt keywords
- category language
- tie-break logic

Matcher should prefer:

- a more specific scaffold over a broad one
- a broad fallback only when confidence is low

EXTERNAL REFERENCE RULE
External templates may be used as:

- technical base
- structure reference
- engineering reference

External templates should NOT be copied wholesale into the scaffold library.

CHANGE RULE
When scaffold behavior or structure changes:

1) Update runtime code first
2) Update this file if the contract changed
3) Update:
   - `LLM/egen-motor/scaffold-status-and-plan.md`
   - `LLM/egen-motor/NEXT-CHAT-HANDOFF.md`
   if priorities or roadmap meaningfully changed

WHEN NOT TO ADD A NEW SCAFFOLD
Do not add a new scaffold if:

- it is only a visual variant of an existing one
- it requires heavy backend setup to be useful
- it can be handled by prompt variation on top of an existing scaffold
- it would mostly duplicate another scaffold's structure

SUCCESS CRITERIA
A scaffold is successful if it improves:

- start quality
- section completeness
- visual baseline
- consistency across generations
- preview readability
- download/build reliability
