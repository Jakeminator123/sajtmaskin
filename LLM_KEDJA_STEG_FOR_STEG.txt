LLM-KEDJAN - STEG FOR STEG
==========================

Detta dokument fylls pa under den stegvisa genomgangen av LLM-kedjan.
Fokus:
- vad ar input
- vad ar output
- vad ar source of truth
- vad ar hjalpdata
- vad far faktiskt na modellen
- vad ar risk / stale / legacy

Samlad slutoversikt over alla fem steg:
- 5-steg.txt


STEG 1 - BUILDER ENTRY / PROMPT IN / FORSTA KLASSNING
-----------------------------------------------------

Primara kodkallor
- docs/schemas/builder-entry-contract.md
- src/app/builder/useBuilderState.ts
- src/app/builder/useBuilderPageController.ts
- src/app/builder/useBuilderEffects.ts
- src/app/builder/useBuilderPromptActions.ts
- src/lib/hooks/chat/useCreateChat.ts
- src/lib/hooks/chat/useChatMessaging.ts

Vad ar input?
- URL-parametrar till /builder
  - buildMethod
  - buildIntent
  - project
  - prompt
  - promptId
  - templateId
  - chatId
  - source=audit (kompat / legacy)
- lokal client state
- project/client persistence
- ev. prompt handoff via /api/prompts/[promptId]
- ev. template-init via /api/template

Vad ar source of truth?
- builder-entry-contract.md for den onskade publika kontraktsbilden
- useBuilderState.ts + useBuilderPageController.ts for faktisk klientlogik

Vad ar output fran steg 1?
- ett aktivt builder-state med:
  - appProjectId
  - buildMethod
  - resolvedBuildIntent
  - chatId eller null
  - resolvedPrompt eller templateId-init
  - en normaliserad entryKind / entryState for ingressen
- detta mynnar senare ut i create chat eller template init

Vad far faktiskt na modellen i detta steg?
- ingenting an
- detta steg klassar och forbereder builder-state
- modellen far forst input nar create-chat / stream-begaran byggs

Bra saker just nu
- fresh-entry guards finns och blockerar stale chat-restore
- prompt-driven och template-driven entry ar dokumenterade separat
- templateId ar tydligt markerad som transient trigger, inte durable ID
- entry-logiken ar nu samlad under builder-entry / entryKind i stallet for att
  spridas lika mycket over flera effekter
- dod instruction-sync till /api/v0/projects/instructions ar borttagen ur
  builderflodet

Risker / komplexitet
- source of truth ar mindre utspridd nu, men det finns fortfarande flera
  kompatlager:
  - hasEntryParams
  - entryIntentActive
  - templateId-specialfall
  - gamla payload-falt som v0ProjectId
- legacy v0ProjectId lever kvar i payload-/DB-kompatibilitet och kan fortsatt
  forvirra om det blandas ihop med appProjectId
- flera ingressvagar samverkar:
  - promptId fetch
  - prompt param
  - template init
  - project restore
  - auto project init

Min bedomning
- Steg 1 ar ganska robust, men mer komplext an det borde vara.
- Ingen akut bug sticker ut just nu efter saneringen.
- Den viktigaste kvarvarande uppgiften ar att fortsatta minska legacy-namn i
  payload/kontrakt utan att bryta gamla data- och kompatspår.

Nasta steg
- granska scaffoldval + BuildSpec + route plan + pre-generation contracts


TVARGAENDE NOTER FOR KOMMANDE STEG
----------------------------------

Detta dokument ska lasas tillsammans med:
- SYSTEMKARTA_SAJTMASKIN.txt
- docs/architecture/builder-generation.md

Nar runtime-sanningen andras i ett steg ska vi samtidigt halla ajour:
- kanoniska docs/schemas
- SYSTEMKARTA_SAJTMASKIN.txt
- config/dashboard/app.py och/eller scripts/scripts_dashboard.py nar deras
  overblick, status eller kartyta annars blir missvisande

Prompt-dumps ar debug/observability, inte source of truth. Delad statuslogik for:
- orchestration-dynamic
- own-engine-codegen
- plan-mode-planner
ligger i scripts/dashboard_shared.py.

Steg 2 / route plan (snabb sanning):
- RoutePlan har provenance.primarySource (brief / prompt / scaffold) och provenance.sources[].
- BuildSpec använder bland annat routePlan.siteType och scaffold-signaler för contextPolicy/qualityTarget.
- Follow-up: persisted scaffold kan låsas upp vid tydlig redesign i auto-läge (shouldIgnorePersistedScaffoldForMatch).

Repo-ratt stegordning for resten av kedjan:
- Steg 1 = normalisera ingressen och las begreppen
- Steg 2 = orkestrera generationen
- Steg 3 = bygg faktisk LLM-input
- Steg 4 = generera, finalisera och validera resultatet
- Steg 5 = materialisera version/preview och folj upp stabilitet

Tvargaende krav i varje steg:
- runtime-koden ar source of truth
- docs, systemkarta och dashboardar ska spegla koden efter verifiering
- minska naming debt, dubbletter och stale spar i samma pass nar det ar sakert

- Var slutar Steg 4 och var börjar preview/tier-2 (Steg 5)?

STEG 5 - PREVIEW START / SESSION / STATUS (SMALT KONTRAKTSPASS)
---------------------------------------------------------------

Kallor:
- docs/architecture/preview-deploy.md
- src/lib/providers/own-engine/generation-stream-post-finalize.ts
- src/lib/hooks/chat/stream-handlers.ts
- src/app/api/engine/chats/[chatId]/preview-session/route.ts
- src/app/api/engine/chats/[chatId]/preview-status/route.ts
- src/lib/gen/preview/preview-session.ts

Vad ar input?
- Sparad version efter finalize (`done` i stream)
- preview-start policy + filesJson
- aktiv sessionstatus (om den finns) for chat+version

Vad ar output?
- Antingen `preview-ready` med faktisk `previewUrl` + `previewSessionId`
- Eller `build-error` med stage/message
- Eller statussvar (`running` / `stopped` / `missing` / `version_mismatch`) for recover

Vad ska vi kunna svara pa?
- Att `done` betyder "version sparad", inte "preview live"
- Att `previewPending` betyder "boot forvantas", inte "preview klar"
- Att `previewUrlHint` ar tillfallig hint under boot, inte persisterad live-url
- Att `preview-session` och `preview-status` ar version-bundna och skiljer `reused_url` / `resumed` / `recreated`
- Att `legacyShimPreviewUrl` ar fallback/shim, inte primar own-engine preview
