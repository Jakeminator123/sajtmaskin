SAJTMASKIN PROJECT STATUS HANDOFF
Generated: 2026-03-10

1. WHAT THIS PROJECT IS

Sajtmaskin is an AI-powered website builder.
The current direction is that the default generation path should use our own
engine locally/in-app, with v0 as fallback only when explicitly enabled.

Core flow:
- user prompt in builder
- own engine generates/updates project files
- local preview renders generated code
- sandbox can run a fuller Next.js runtime when needed
- finished version can be downloaded or deployed

4. KEY LESSONS LEARNED

- Preview-specific hacks must never mutate code that gets saved, downloaded, or executed elsewhere.
- TypeScript passing is not enough; Next.js App Router build rules can still fail production builds.
- Preview success is not the same as true runtime success.
- Stateful error memory makes autofix much more useful than stateless retries.


5. THINGS THAT ARE STILL TRUE / LIMITATIONS

6b. LATEST FOLLOW-UP AFTER HANDOFF FILE WAS CREATED

- The manual design-system picker in the builder header was removed from the
  main UI because it was low-signal compared to the space/attention it took.
- Underlying support is still present in code/state, so this can later return
  as a smarter prompt-driven or advanced-only control.
- `.env.local` / `.env.production` were lightly cleaned:
  - removed dead `MCP_GENERATED_CODE_API_KEY`
  - removed duplicated trailing `OPENAI_API_KEY`
  - removed obviously unused `HUGGINGFACE_API_KEY`
  - removed dead experimental model flags from local env
- Current working theory for Unsplash/image confusion:
  - image generation currently happens in two phases
  - phase 1: placeholder -> direct Unsplash URL
  - phase 2: optional Blob materialization/upload
  - if Blob materialization is disabled or not available, images may still work
    visually as Unsplash URLs but not end up persisted the way the user expects
- Blob storage is namespaced per chat/version in code:
  `images/{chatId}/{versionId}/{hash}.{ext}`
- So the likely problem is not "all generations share one blob bucket path", but
  rather when the Blob step runs, whether it runs at all, and whether the final
  version content is refreshed after replacement.
- The likely next engineering task is to unify image handling into one clearer
  pipeline:
  placeholder -> choose image -> if blob enabled upload immediately -> persist
  final URL to version
