Ni har redan ett lager som kan upptäcka integrationer i genererad kod och visa dem som integrationskort / env-var-prompts i klienten. detect-integrations.ts letar efter env-vars, SDK-importer och kända mönster för bland annat Supabase, Stripe, Clerk, Auth.js, Prisma, OpenAI, Vercel Blob/KV och Google OAuth/Analytics.

Min huvudrekommendation

Tänk i tre nivåer, inte en:

1. Preview utan riktiga integrationer

Det här ska vara standard.

När användaren bara vill:

se layouten
känna känslan
testa sidorna
se navigation, copy, sektioner och komponenter

då ska sidan kunna köras med:

mockdata
fejkad auth-state
disabled submit-knappar
lokala placeholders
“connect later”-kort

Det här är den bästa defaulten.

För de flesta användare är det mycket bättre än att tvingas fylla i:

OAuth-nycklar
databas-URL
Stripe keys
API-hemligheter

innan de ens vet om de gillar sidan.

Och ditt repo stödjer redan det tänket genom dataMode och unresolved contract-flödet.

2. Managed preview / sandbox för “nästan på riktigt”

Det här är ett mellanläge.

Om användaren vill kunna uppleva något mer verkligt innan publicering, kan du erbjuda ett kontrollerat testläge där vissa integrationer körs via:

delade testnycklar
sandbox-konton
demo-databas
icke-kritiska mockade callbacks

Det här är mest användbart för:

auth-flöden
formulär med sparande
dashboard-demo
enkla CMS-liknande upplevelser
betalflöden i test mode

Men jag hade inte gjort detta till default för alla sajter.

Varför:

det blir dyrt
det blir mer drift
det blir säkerhets- och isolationsfrågor
och du måste städa mycket tillfällig state

Så:
sandbox är bra som premium-/advanced-läge, inte som grundkrav.

3. Riktiga integrationer först vid publish / connect

Det här är där riktiga nycklar ska in.

Det vill säga:

användaren är nöjd med sidan
vill deploya
då får de en lista på vad som krävs för att denna sajt ska fungera fullt ut

Här passar ditt nuvarande detect-/contract-lager väldigt bra:

“den här sajten kräver DATABASE_URL”
“den här använder Clerk”
“den här behöver GOOGLE_CLIENT_ID och GOOGLE_CLIENT_SECRET”
“den här använder Stripe”

Det gör att publish-steget kan bli:

analysera appen
visa vilka integrationer som krävs
låt användaren koppla dem
deploya

Det är ett mycket bättre UX än att fråga om allt i början.

Så ska du tänka kring OAuth specifikt

OAuth är ett av de tydligaste fallen där du inte vill kräva full setup direkt.

För preview räcker det ofta med:

dummy login state
fake user menu
auth forms som ser riktiga ut
tydlig markering “koppla Google/GitHub vid publicering”

För riktig funktion behöver du sedan:

client ID / client secret
rätt callback URL
rätt domän
ofta produktionsgodkänd appkonfiguration

Det gör OAuth till en perfekt kandidat för:
“simulate in preview, configure on publish.”

Behöver du hyra sandboxes?

Inte för allt.

Jag hade bara tänkt sandboxes för vissa fall.

Du behöver sannolikt inte sandboxes för:
landningssidor
blogs
content sites
portfolios
enklare e-commerce-frontends utan riktig checkout
dashboards där data kan vara demo
Du kan vilja ha sandboxes för:
riktig auth-preview
riktig databas-persistens i preview
AI-funktioner som kräver server-side keys
webhook-/OAuth-testning
betalflöden i test mode

Så svaret är:
nej, du måste inte köpa/hyra sandboxes som grundmodell.
Börja utan det.

Kan de publicera via dig till Vercel?

Ja, det är faktiskt ofta den smartaste första modellen.

Och ditt repo tyder redan på att du har tänkt i den riktningen. En plan i repot beskriver ett deploysteg där backend redan finns för domänsökning, domänlänkning och pris via Vercel, och att användaren efter lyckad deploy ska kunna söka, köpa eller koppla egen domän i samma flöde.

Det tyder på att modellen:

Sajtmaskin deployar
sedan kopplas domän
sedan blir sidan live
