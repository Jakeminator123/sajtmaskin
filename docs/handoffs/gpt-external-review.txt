Repository reality check
Your repo contains two different meanings of “v0”, and conflating them is a major source of confusion when debugging “why am I seeing static pages?”:

One “v0” is simply your internal API namespace (/api/v0/...) used by the builder UI to talk to your own backend (chat streams, files, sandbox preview, etc.). It is not the external v0 Platform. This internal API is what powers the own-engine pipeline and the sandbox bootstrap logic in the builder.

The other “v0” is the external v0 Platform integration, implemented under src/lib/v0/ (e.g. v0-sdk usage) and routes that explicitly initialize templates/registry via that platform. The v0-generator.ts header comment is explicit that own-engine builder streams do not use the v0 generator module; it is reserved for template/registry/ZIP flows that still point at v0 Platform.


On master you also already have an explicit “keep own-engine clean” policy enforced by a boundary test: src/lib/own-engine and src/lib/providers/own-engine must not import v0 Platform internals or the v0 SDK. That’s a strong foundation for gradually removing v0 without regressing your own-engine path.

Current own-engine pipeline as implemented today
At a code level, the own-engine flow is not “one monolithic LLM call.” It is a deterministic orchestration → stream generation → finalize pipeline → preview tiering pipeline—where the “static preview” is intentionally a tier-1 shim and the full Next runtime is a tier-2 sandbox that comes later.

Where “awaiting input” fits (and why it doesn’t “continue”)
You asked earlier whether “Awaiting input” should loop back into the chain once the user answers. In your code, it does—by design.

On follow-ups, the stream route explicitly:

infers contracts,
asks a clarification question if required (returning a contract-gate SSE stream),
later consumes the user’s reply as a contract answer, and then continues generation using confirmed decisions.
So the correct diagram is:

If you want the UI to more explicitly “re-enter the chain,” you should treat the user’s answer as a first-class event (“contract.answer.submitted”) and show a resumed pipeline state in the progress strip—today the resume does happen, but it can feel implicit.


Why you’re seeing “static pages” instead of a full Next sandbox runtime
Based on the code on master, you are not getting a static preview because a “static production wins a race.” You are getting tier-1 shim preview because the system intentionally returns it first, and tier-2 sandbox preview is either not starting, failing, or not being upgraded.

The shim is a real, intentional tier
The route /api/preview-render serves a self-contained HTML preview built from the generated files. It’s explicitly described as a fast/legacy path and is not a full Next runtime.

That route even warns that it needs a renderable default-export React component (e.g. app/page.tsx) and points users to sandbox preview for “full Next.js running.”

The server intentionally emits shim first—even when sandbox will run
Your own-engine generation stream emits done with demoUrl = shimUrl while sandbox starts in the background; sandbox success/failure is communicated later via sandbox-ready or build-error.


On the client side, the SSE consumer explicitly upgrades currentDemoUrl to the sandbox URL only once it receives sandbox-ready.

The builder also tries to “upgrade” older versions automatically
Even if a generation finished earlier and you reopened the builder later, useBuilderPageController will attempt to bootstrap a sandbox preview when:

you are authenticated,
it’s an own-engine chat (not a v0-style chat),
the active version doesn’t already have a stored sandbox URL,
and the current preview URL is tier-1 (shim) or unset.
If that POST to /api/v0/chats/{chatId}/sandbox-preview never succeeds, you’ll stay on tier-1.


The most common concrete causes in your code
The following are the “real reasons” (not guesses) why tier-2 is not appearing:

Sandbox is not configured. Your sandbox route returns a dedicated sandbox_disabled style error when sandbox credentials are missing, and you ship a setup hint that explicitly names the supported credential strategies (OIDC token or access token + team/project).


Authentication is required for the builder’s sandbox bootstrap effect. The upgrade logic in the builder exits early unless isAuthenticated is true. If you’re testing logged out (or auth is flaky locally), you will reliably stay on the shim.

Sandbox can be blocked after finalize. The server-side sandbox kickoff is gated on three booleans: sandbox configured, preview not blocked, and parsed file count > 0. If preflight marks the version as preview-blocked, sandbox won’t run in the generation stream.



Sandbox boot fails in install/dev/build. The sandbox runtime calls npm install, then starts npm run dev, and optionally runs npm run build. Any failure is surfaced as a typed error and in the SSE contract as build-error, after which the UI falls back to shim and shows a toast.



Generated projects exceed sandbox limits. The runtime enforces hard caps (file count and total bytes) and will throw before sandbox startup if exceeded. This produces “I only see shim” without necessarily meaning generation failed.

One subtle code-quality issue worth fixing immediately: resolveSandboxPreviewModeFromEnv() defaults to dev_then_build, but the SandboxRuntimeOptions docstring in the same module claims the default is dev_only. That contradiction is exactly the kind of “docs vs reality” mismatch that causes operator confusion during debugging.

v0 dependency audit and a clean removal plan
If your product goal is “own-engine only, no v0 folder, no v0 Platform dependency,” you need to separate “what’s already independent” from “what’s still v0-backed.”

What is already own-engine and does not require v0 Platform
The core own-engine generation + finalize pipeline is fully independent:

Finalize includes deterministic autofix, esbuild validation + optional LLM fixer, URL expansion, image materialization, merge, preflight, and persistence into your Postgres-backed chat/version store.


Sandbox runtime is also not v0 Platform–dependent; it relies on Vercel Sandbox credentials and @vercel/sandbox, not v0 Platform.

ZIP download for own-engine versions is already implemented against your engine version storage (engine_versions.files_json).


What is still backed by v0 Platform (and therefore blocks “no v0 anywhere”)
Template initialization currently calls generateFromTemplate from the v0 generator (v0-sdk), caches template results keyed by user, and persists rows keyed by v0 chat/version IDs. If you want “own-engine only,” this entire path must be replaced or removed.


There are still shared utilities living under src/lib/v0/ that are being imported from own-engine endpoints (e.g. normalizeV0Error). That’s not a runtime dependency on v0 Platform, but it is a repo-level dependency that prevents deleting the v0 folder.


The src/lib/v0.ts wrapper documents that v0 SDK is still used for certain platform features and that a “v0 fallback builder” flag affects preview URL preference when a v0-hosted demo exists. If your policy is “never prefer v0 demos,” you should remove that branch entirely.


A pragmatic removal sequence that won’t break the product
The cleanest way to remove v0 without destabilizing the builder is to treat it as a migration with explicit “done” milestones:

First, make “own-engine only” a runtime mode.

Add a single feature flag like SAJTMASKIN_V0_PLATFORM_ENABLED=false and make every v0 Platform entry point early-return with a clear error. This avoids “half-working” states while you migrate templates and registry init.

Second, relocate generic utilities out of src/lib/v0/.

For example, move normalizeV0Error into src/lib/providers/errors/normalize-provider-error.ts (or similar), and rewire imports in /api/v0/chats/*/stream so these routes no longer import anything from a v0 folder. Your own-engine boundary test already formalizes this direction for core modules.


Third, replace v0-backed templates with own-engine scaffolds + internal template library.

The fastest replacement pattern is:

treat “template selection” as selecting an internal scaffold + reference library snippet set,
then run an own-engine “template hydration” pipeline step that produces the project by applying your system prompt + brief/spec rules to that scaffold (instead of cloning v0 output).
This aligns with how your own-engine already merges generated files against a scaffold base in finalize.

Fourth, deprecate v0 ID fields in your persistence layer.

Today your template route persists v0ChatId, v0ProjectId, v0VersionId etc. If you remove v0, you should migrate schemas to store only engine IDs, otherwise you’ll keep “v0-shaped” data models forever.

Fifth, delete src/lib/v0/ and remove v0 Platform routes.

At this point you can safely remove v0-sdk from dependencies and delete the v0 folder. The boundary test can be upgraded to: “no src/lib/v0 folder exists on disk” (hard fail), which ensures the repo stays clean.

A world-class target: LLM flows + repair loops + tests + sandbox preview
The good news is that you already have several “world-class” ingredients:

structured finalize pipeline (autofix + validation fixer loop + preflight),
sandbox preview machinery with readiness probing and optional build verification,
post-generation checks that run image validation and then kick off a sandbox “quality gate” (typecheck + build) and can queue autofix if those checks fail.
So the “world-class plan” is less about replacing everything and more about turning what you already have into an explicit, measurable loop where visual quality + correctness are enforced, not just requested in prompts.

Concrete improvements you should implement next
Make “sandbox is missing” impossible to miss.

Right now, if sandbox isn’t configured you can get stuck on shim and interpret it as “the engine is only producing static.” You already have a strong setup hint string for operators; surface it in the UI whenever the sandbox bootstrap endpoint returns sandbox_disabled (not only in dev).


Make correctness gates consistent and explicit.

You already run a sandbox quality gate (typecheck + build) from post-checks. The key upgrade is to make the result feed back into generation deterministically:

if typecheck/build fails, trigger a “Build Engineer” repair loop that edits code/package.json minimally,
re-run gates,
only then mark a version “world-class ready.”
Reduce dependency drift in sandbox.

Your sandbox runtime uses npm install and a git template repo URL (defaulting to vercel/sandbox-example-next). For reliability:

pin the template repo URL to a commit or your own fork via SAJTMASKIN_SANDBOX_TEMPLATE_GIT_URL,
move toward npm ci by ensuring a lockfile is generated and included in the sandbox project build step,
implement an “ERESOLVE repair agent” that proposes controlled version pins and retries install.
Make visual excellence measurable.

You already have an inspector system in the builder UI (element mapping / Playwright capture paths are implied by UI features), which is a perfect foundation for an automated “Design QA agent.” The missing step is simply to run a post-sandbox screenshot + heuristic evaluation and then apply fixes automatically before the user ever sees the final preview.
