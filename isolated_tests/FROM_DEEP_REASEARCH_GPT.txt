Current own‑engine pipeline as implemented on master
Your “own‑engine” path on master is already a real end‑to‑end system: prompt orchestration → scaffold selection (keyword + embeddings fallback) → dynamic system prompt assembly → generation stream → deterministic autofix/repair → version persistence → sandbox verification/preview.

Prompt intake and orchestration (create and follow‑up)
Both the “new chat” route and the “follow‑up message” route are served under /api/v0/chats/..., but these routes are executing own‑engine behavior (not external v0 Platform generation). The create route does prompt orchestration, resolves a model tier, prepares orchestration context, and then calls the own‑engine pipeline stream creator.

The follow‑up route is more advanced: it can inject orchestration continuity into follow‑ups, classify whether the user is requesting a “clear redesign” vs a small edit, and it builds a file context based on the last version’s files to support “edit instead of regenerate.”

Scaffold matching and “dossier-like” enrichment
Scaffold selection is deterministic keyword scoring first, with embedding search only as a fallback when a generic scaffold would otherwise be selected. Guardrails prevent embeddings from incorrectly upgrading a generic website into an “auth/app” scaffold unless the prompt actually signals it.

Only scaffolds listed in the internal registry are eligible at runtime; the registry explicitly states that external template research is reference material and not runtime input.

On top of scaffolds, your system prompt builder also injects “template references” via a curated template library (scored references + optional extracted code snippets) and pulls “knowledge base” matches with optional semantic fallback. That’s the closest thing you already have to “dossiers.”

System prompt assembly (static core + dynamic context)
Your prompt architecture is already close to best practice:

Static core is loaded from workspace config and intended to stay stable for prefix caching.
Dynamic context carries build intent rules (website/template/app), scaffold + route plan + contracts, plus imagery and SEO rules, plus docs/template-library injections.
This is a strong foundation for “world-class by default,” because it establishes consistent design constraints and quality expectations at generation time.

Sandbox verification as a first‑class gate

This is exactly the type of objective correctness gate that separates “pretty in iframe” from “real Next project that actually builds.”

Concrete issues, bugs, and semantic debt that still block “atomic clarity”
You asked explicitly for “buggigt / brister” and semantic cleanup, with one exception: keep v0-templates intact but clearly separated.

Below are the highest-leverage issues visible from code.

Semantic inconsistency: imagery rules vs Next image host allowlist
Your system prompt’s imagery rules explicitly forbid picsum.photos and similar placeholder services and instruct the model to use /placeholder.svg?... for all images.

But your generated next.config.ts scaffold currently allows remote patterns that include picsum.photos and via.placeholder.com.

This mismatch is a quality leak: even if the prompt says “never,” the runtime config still allows it, increasing the chance the model (or a later fixer) accidentally uses lower-quality imagery sources.

Duplication drift: sandbox template URL and path safety checks
You have a central sandbox runtime module that supports overriding the git template source via SAJTMASKIN_SANDBOX_TEMPLATE_GIT_URL and exposes shared helpers like isSafeRelativePath.

But the quality-gate route creates its own sandbox directly and hardcodes the template repo URL (https://github.com/vercel/sandbox-example-next.git) and implements its own isSafeRelativePath variant.

This is not immediately “broken,” but it is classic atomicity debt: two implementations of “sandbox source and path rules” will drift and cause confusing behavior differences (example: quality gate VM differs from preview VM).

“dev_then_build” naming vs reality in sandbox preview runtime
Your sandbox runtime module defaults SAJTMASKIN_SANDBOX_PREVIEW_MODE to dev_then_build, but still treats actual build verification as opt-in via options.verifyBuild === true.

That is an easy-to-misunderstand semantic trap: operators will assume “dev_then_build default” implies the build runs by default, when in code it does not unless the caller sets verifyBuild.

Your new quality gate route does run next build by default (good), but the preview runtime path still has this ambiguity.


Baseline scaffold dependency size risk
Your export scaffold baseline package.json includes many large dependencies by default (including 3D libraries), even if the generated site doesn’t use them.

This is a reliability risk for sandbox installs (slow installs, higher likelihood of transient install failures). It may have been chosen to enable “world-class visuals,” but the tradeoff is real, and it’s exactly the kind of thing that causes “sandbox never upgrades, stuck on shim” in practice.

Visual QA exists, but is still static and not screenshot-based
The code includes a Visual QA module behind SAJTMASKIN_VISUAL_QA=1.

However, it does not actually capture screenshots; analyzeVisualQuality() performs heuristic checks on source files and returns screenshotCaptured: false.

This is a gap relative to your “world-class in the sandbox iframe” goal. Your current gate verifies build correctness in a VM (great), but doesn’t yet verify what it looks like as rendered.

“v0‑templates” should remain, but must become an explicitly isolated subsystem
The script scripts/sync-v0-templates.mjs explicitly fetches templates from v0.app and regenerates src/lib/templates/templates.json and src/lib/templates/template-categories.json.

That is fine to keep as “v0-templates,” but to achieve your goal (“everything else is own-engine”), you need a clear boundary so nothing else in own-engine implicitly depends on v0 concepts beyond this catalogue.

A world-class target architecture that preserves v0‑templates but makes everything else own‑engine
The “atomic” end state that matches your request is:

One canonical generation pipeline (own-engine).
One canonical artefact builder (now you have buildExportableProject—great).
One canonical correctness policy (quality gate + retry loop).
One isolated “v0 templates catalogue” subsystem.
A practical target decomposition:

Own-engine runtime core

/src/lib/own-engine/** = orchestration session, generation stream, finalize, and upgrade/repair loops
/src/lib/gen/** = reusable generation utilities (scaffolds, system prompt, routing, contracts, parsing)
/src/lib/mcp/runtime-url.ts = sandbox runtime primitives as the shared sandbox API surface
v0-templates isolated

Keep scripts/sync-v0-templates.mjs and the JSON outputs untouched in behavior.
Add a small “template catalogue boundary” layer so the rest of the app imports only from one stable module (example: src/lib/v0-templates/catalog.ts which internally reads those JSON files). The script stays as-is; only the consuming imports become more explicit.
API semantics

Introduce /api/engine/** as the canonical own-engine API routes.
Keep /api/v0/** only as compatibility wrappers (or as “API versioning only”), but stop calling own-engine “v0” in naming.
This resolves the user-facing confusion: “v0 still exists everywhere,” while still retaining a stable API version namespace if you want it.
