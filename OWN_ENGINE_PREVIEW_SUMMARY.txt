Own-engine preview summary
==========================

Short answer:
The default own-engine preview is not a full Node.js build of the generated app.
It is a fast internal preview surface that renders a self-contained HTML view from
the saved version files. That is why it can feel "free" compared to a full runtime,
but it is not the same as running npm install + npm run dev/build for each version.

What actually happens in default own-engine preview:
- The builder sends the prompt into the own-engine lane.
- The engine generates code and saves a version in Postgres.
- The preview route loads the saved files and builds a self-contained HTML preview.
- The iframe shows that HTML as the primary iteration surface.

What this means:
- You do get a preview without using v0.
- You do not get a full Next.js/Node.js runtime by default.
- The preview is cheaper because it avoids booting a full sandbox for every generation.
- The preview is useful for fast iteration, but it is not identical to a deployed runtime.

When you do get something closer to a real Node.js runtime:
- That happens through sandbox, not through the normal own-engine preview.
- Sandbox creates a runtime, writes files, runs npm install, and starts npm run dev.
- This is much closer to a real app runtime than the standard preview.

Cost reality:
- There is no separate v0 preview cost in own-engine mode.
- You still pay for your own model/API usage.
- You still pay for your own app/platform infrastructure.
- Sandbox and deploy are not truly free; they consume more compute than the HTML preview.

Is v0 involved when fallback is off?
- In the actual own-engine generation and default preview flow, it appears no.
- v0 fallback is only used when the fallback flag and explicit fallback path are active.
- The fact that some routes still live under /api/v0/... does not mean v0 is active in the own-engine lane.

Best one-paragraph explanation:
In own-engine mode, Sajtmaskin does not rely on v0 to generate or preview the site.
It generates code itself, stores version files in Postgres, and renders a fast preview
through its own preview-render route. That preview is not a full Node.js build of the
generated app, but a cheaper self-contained runtime surface for iteration. If you want
real Node.js/Next.js runtime behavior, you need sandbox or a real deployment, which
uses more compute and is therefore not meaningfully "free".
